# PR ワークフロー

ユーザーが PR 準備を任せたいとき、またはブランチ作成、ステージング、コミット、push まで Codex にやってほしいと言ったときに使います。

## 基本方針

git 作業を主タスクとして扱います。実装が終わったら、ユーザーから止める指示がない限り、リモートへ push された状態まで持っていきます。

## 手順

1. `git status --short --branch` と `git branch --show-current` で状態を確認します。
2. 現在のブランチが `main` なら、ステージング前に新しい作業ブランチを切ります。`main` / `master` 以外でも、明らかに一時的、検証用、誤操作由来に見える不自然なブランチ名にいる場合は、そのブランチを使い続けず `main` に戻してから新しい作業ブランチを切ります。
3. 変更内容に合わせて短く分かりやすいブランチ名を付けます。`feat/`、`fix/`、`docs/`、`chore/` などの接頭辞を優先します。
4. 差分を確認し、今回の依頼に含まれるファイルだけをステージングします。基本的に `git add .` を行うが、明らかにおかしなファイルがステージングされた場合ユーザーに警告、確認を行う。
5. リポジトリの流儀に合う、短いコミットメッセージを作ります。
6. `git push -u origin <branch>` で upstream tracking 付きで push します。
7. 最後に、ブランチ名、コミットメッセージ、`origin` に push 済みかを報告します。

## 安全策

- ユーザーが PR 準備を任せた場合、作業を `main` や `master` に置いたままにしません。
- 明示的に求められない限り、履歴の書き換え、過去コミットの amend、破壊的な git コマンドは使いません。
- 作業ツリーに無関係な変更が混ざっていて安全に切り分けられない場合は、全部をステージングせず、衝突内容を説明して止まります。
- すでに作業ブランチ上にいる場合は、ユーザーから新しいブランチ指定がない限り、そのまま使います。
- ただし、`tmp`、`test`、`backup`、`wip` のような明らかに正式作業向きでないブランチ名や、意図が読み取れない不自然なブランチ名なら、そのまま続行しません。`main` に戻れることを確認したうえで、新しい作業ブランチを切ります。
- push が認証、権限、ネットワークなどで失敗した場合は、試したコマンドと具体的な失敗理由を報告します。

## PR 作成が必要な場合

ユーザーが PR 作成そのものも明示的に依頼し、環境が対応している場合は、push のあとに次へ進みます。

1. 差分を要約し、分かりやすい PR タイトルと本文を作ります。
2. 作成したタイトルと本文で `gh pr create` を実行します。
3. 作成された PR の URL を返します。
